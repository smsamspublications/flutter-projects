PART II — THE FLUTTER FRAMEWORK
From widgets to shipped applications

Unit 18Introduction to Flutter

Flutter is Google's open-source UI toolkit for building natively compiled applications for mobile, web, and desktop from a single Dart codebase. Unlike frameworks that wrap native platform widgets, Flutter draws every pixel itself using its own high-performance rendering engine (based on the Impeller/Skia graphics engines), which is why a Flutter button looks and behaves identically on Android, iOS, web, and desktop.

How Flutter Renders a Screen

Everything you see on screen in Flutter is a widget — and widgets themselves are lightweight, immutable configuration objects, not the actual pixels. When something changes, Flutter doesn't repaint the whole screen; it rebuilds only the affected part of a tree of three parallel structures:

  • Widget tree — the declarative description of what the UI should look like (what you write).
  • Element tree — the mutable, live instantiation that manages a widget's lifecycle and links a widget to its position in the tree.
  • Render tree (RenderObject) — the layer that actually handles layout, painting, and hit-testing.
TIPYou will spend almost all of your time working with the widget tree; Elements and RenderObjects work behind the scenes and you rarely touch them directly unless you're building highly custom, low-level widgets.

Flutter vs Other Approaches

ApproachExampleHow UI is rendered
NativeSwift/KotlinPlatform's own native widgets, separate codebase per platform
WebView-based hybridCordova/IonicHTML/CSS/JS rendered in an embedded browser
Native-bridgeReact NativeJavaScript logic calls out to real native widgets via a bridge
FlutterDart + FlutterFlutter's own engine draws every pixel directly, one codebase, no bridge
KEY POINTS
  • Flutter draws its own UI rather than wrapping native platform widgets, giving pixel-perfect consistency across platforms.
  • "Everything is a widget" — text, padding, layout, even animations are all expressed as widgets.
  • The Widget → Element → RenderObject pipeline is what makes Flutter fast: only the parts of the tree that actually changed are rebuilt and repainted.

Running flutter create scaffolds a standard folder structure. Understanding what each folder is for will save you a lot of confusion later.

The Default main.dart

import 'package:flutter/material.dart';

void main() {
  runApp(const MyApp());
}

class MyApp extends StatelessWidget {
  const MyApp({super.key});

  @override
  Widget build(BuildContext context) {
    return MaterialApp(
      title: 'My First App',
      theme: ThemeData(primarySwatch: Colors.blue),
      home: const HomePage(),
    );
  }
}

runApp() takes a widget and makes it the root of the entire widget tree, attaching it to the screen. MaterialApp is a convenience widget that sets up Material Design theming, navigation, and localization for the whole app in one place.

Unit 21Widgets: The Building Blocks

There are two fundamental categories of widget in Flutter, and choosing the right one for each piece of UI is one of the first real design decisions you'll make in any screen.

StatelessWidget

A StatelessWidget describes UI that never changes once built — it has no internal, mutable data. It is rebuilt only when its parent passes it new configuration.

class Greeting extends StatelessWidget {
  final String name;
  const Greeting({super.key, required this.name});

  @override
  Widget build(BuildContext context) {
    return Text('Hello, $name!');
  }
}

StatefulWidget

A StatefulWidget can hold data that changes over time (a counter, a toggled switch, text typed by the user) and rebuild itself whenever that data changes, via setState(). Every StatefulWidget is paired with a separate State object that actually holds the mutable data — the widget itself stays immutable.

class Counter extends StatefulWidget {
  const Counter({super.key});

  @override
  State<Counter> createState() => _CounterState();
}

class _CounterState extends State<Counter> {
  int _count = 0;

  void _increment() {
    setState(() {
      _count++;
    });
  }

  @override
  Widget build(BuildContext context) {
    return Column(
      children: [
        Text('Count: $_count'),
        ElevatedButton(onPressed: _increment, child: const Text('Add')),
      ],
    );
  }
}
CAUTIONNever mutate state fields directly (_count++; on its own) — Flutter has no way to know the UI needs rebuilding unless the change happens inside setState().

The StatefulWidget Lifecycle

MethodCalled when
initState()Once, when the State object is first created — good place for one-time setup (subscriptions, controllers)
build()Every time the widget needs to (re)render
didUpdateWidget()When the parent rebuilds this widget with new configuration
setState()Called by your own code to mark state as changed and trigger a rebuild
dispose()Once, when the State object is permanently removed — release controllers, cancel subscriptions here

BuildContext

Every build() method receives a BuildContext — a handle to the widget's location in the tree. It's used to look up inherited data (theme, media query, navigation) from ancestor widgets, e.g. Theme.of(context) or Navigator.of(context).

KEY POINTS
  • StatelessWidget for UI that never changes on its own; StatefulWidget for UI with mutable, internal data.
  • setState() is the only correct way to tell Flutter that a State object's data changed and the UI needs rebuilding.
  • initState() sets up one-time resources; dispose() releases them — always pair the two.
  • BuildContext is how a widget looks upward in the tree for shared data like theme or navigation.

Unit 22Core Layout Widgets

Flutter has no CSS — layout is built entirely by nesting widgets. A handful of layout widgets, learned well, cover the vast majority of screens you'll ever build.

Text, Container & Image

Row arranges children horizontally; Column arranges children vertically. Both accept mainAxisAlignment (alignment along the direction they lay out) and crossAxisAlignment (alignment perpendicular to it).

      mainAxisAlignment: MainAxisAlignment.spaceBetween,
      children: const [
        Icon(Icons.school),
        Text('LBRCE'),
        Icon(Icons.star),
      ],
    ),
  ],
)

KEY POINTS
  • There is no CSS in Flutter — every layout decision is a widget you nest.
  • Row/Column plus mainAxisAlignment/crossAxisAlignment cover most simple layouts.
  • Stack + Positioned is how you overlap widgets, e.g. a badge on an icon.
  • SizedBox is the idiomatic way to add fixed empty space between widgets.

Unit 23Advanced Layouts & Scrolling

Expanded & Flexible

Inside a Row or Column, Expanded forces a child to fill the remaining available space; Flexible lets a child shrink to fit without necessarily filling all remaining space. The flex property distributes space proportionally between multiple Expanded children.

Row(
  children: [
    Expanded(flex: 2, child: Container(color: Colors.red, height: 50)),
    Expanded(flex: 1, child: Container(color: Colors.green, height: 50)),
  ],
)
// The red box takes twice the width of the green box.

ListView — scrollable lists

// Fixed list of children
ListView(
  children: const [
    ListTile(leading: Icon(Icons.book), title: Text('Data Structures')),
    ListTile(leading: Icon(Icons.code), title: Text('Flutter Lab')),
  ],
)

// Efficient for long or dynamic lists — builds items lazily as they scroll into view
ListView.builder(
  itemCount: subjects.length,
  itemBuilder: (context, index) {
    return ListTile(title: Text(subjects[index]));
  },
)
TIPAlways prefer ListView.builder over a plain ListView(children: [...]) once the list can grow large — .builder only constructs the widgets currently visible on screen, which is dramatically more memory-efficient.

GridView — grid layouts

GridView.builder(
  gridDelegate: const SliverGridDelegateWithFixedCrossAxisCount(
    crossAxisCount: 2,
    crossAxisSpacing: 8,
    mainAxisSpacing: 8,
  ),
  itemCount: 6,
  itemBuilder: (context, index) => Card(
    child: Center(child: Text('Item $index')),
  ),
)

Wrap — flowing layout

Wrap(
  spacing: 8,     // horizontal gap between items
  runSpacing: 8,  // vertical gap between wrapped lines
  children: [
    Chip(label: Text('AI')),
    Chip(label: Text('Data Science')),
    Chip(label: Text('Flutter')),
    Chip(label: Text('IoT')),

)

ListView.builderLong or dynamic vertical/horizontal lists
GridView.builderPhoto grids, dashboards, card layouts

KEY POINTS
  • Expanded/Flexible distribute remaining space inside a Row/Column; flex controls the ratio.
  • .builder constructors (ListView.builder, GridView.builder) are essential for performance once a list can grow large.
  • Wrap is the go-to widget for tag/chip-style content that should flow across lines instead of overflowing.

Unit 25Navigation & Routing

Flutter manages screens with a stack of routes, managed by the Navigator widget — pushing a route shows a new screen on top; popping removes it and reveals what was underneath.

Basic Navigation with push/pop

// Navigate to a new screen
Navigator.push(
  context,
  MaterialPageRoute(builder: (context) => const DetailsPage()),
);

// Go back to the previous screen
Navigator.pop(context);

Passing Data Forward

Navigator.push(
  context,
  MaterialPageRoute(
    builder: (context) => DetailsPage(studentName: 'Anita'),
  ),
);

class DetailsPage extends StatelessWidget {
  final String studentName;
  const DetailsPage({super.key, required this.studentName});

  @override
  Widget build(BuildContext context) {
    return Scaffold(body: Center(child: Text('Hello, $studentName')));
  }
}

Returning Data Backward

// On the calling screen — await the result
final result = await Navigator.push(
  context,
  MaterialPageRoute(builder: (context) => const SelectionPage()),
);
print('User selected: $result');

// On the SelectionPage — pop with a value
Navigator.pop(context, 'Option A');

Named Routes

For apps with many screens, named routes centralize navigation in one place instead of importing every page's class everywhere.

MaterialApp(
  initialRoute: '/',
  routes: {
    '/': (context) => const HomePage(),
    '/details': (context) => const DetailsPage(),
    '/settings': (context) => const SettingsPage(),
  },
)

// Navigate using the route name
Navigator.pushNamed(context, '/details');
TIPFor apps with complex, deep-linkable navigation (especially on the web, where URLs matter), consider the go_router package — it builds on top of Navigator 2.0 and is the community-standard solution for declarative, URL-based routing.
KEY POINTS
  • Navigator.push/pop manipulate a stack of screens; the top of the stack is what's visible.
  • await Navigator.push(...) lets a screen receive a result back once the pushed screen calls Navigator.pop(context, result).
  • Named routes centralize route definitions in MaterialApp, useful once an app has more than a handful of screens.

Unit 26State Management

"State management" is simply the question: when data changes, how does the right part of the UI find out and rebuild? setState() (Unit 15) answers this for a single widget's own local state. As an app grows, you need patterns for sharing state *across* widgets that aren't directly related in the tree.

setState() — Local State

Best for state that belongs to, and is only needed by, one widget and its direct children — a single form's validity, whether a card is expanded, an animation's progress.

InheritedWidget — Flutter's Built-in Sharing Mechanism

InheritedWidget lets any descendant efficiently read data from an ancestor without it being passed down manually through every constructor ("prop drilling"). Most higher-level state management packages (including Provider) are built on top of this primitive.

Provider — the recommended starting point

provider is the package the Flutter team itself recommends for small-to-medium apps. It wraps InheritedWidget in a much friendlier API.

// 1. Define your state as a ChangeNotifier
class CounterModel extends ChangeNotifier {
  int _count = 0;
  int get count => _count;

  void increment() {
    _count++;
    notifyListeners(); // tells every listening widget to rebuild
  }
}

// 2. Provide it above the widgets that need it
void main() {
  runApp(
    ChangeNotifierProvider(
      create: (context) => CounterModel(),
      child: const MyApp(),
    ),
  );
}

// 3. Consume it anywhere below
class CounterDisplay extends StatelessWidget {
  const CounterDisplay({super.key});
  @override
  Widget build(BuildContext context) {
    final count = context.watch<CounterModel>().count;
    return Text('Count: $count');
  }
}

class IncrementButton extends StatelessWidget {
  const IncrementButton({super.key});
  @override
  Widget build(BuildContext context) {
    return ElevatedButton(
      onPressed: () => context.read<CounterModel>().increment(),
      child: const Text('+1'),
    );
  }
}
TIPcontext.watch<T>() subscribes the widget to rebuilds whenever T calls notifyListeners(). context.read<T>() grabs the value once, without subscribing — use it inside callbacks like onPressed where you don't want a rebuild every time you merely read the value.

Riverpod — Provider's Compile-Safe Successor

Riverpod keeps Provider's core idea but removes its two biggest pain points: providers no longer need a BuildContext to be read (so they work in plain Dart, not just widgets), and mismatched provider types are caught by the compiler instead of at runtime.

// 1. Declare a provider at the top level of a file
final counterProvider = StateNotifierProvider<CounterNotifier, int>(
  (ref) => CounterNotifier(),
);

class CounterNotifier extends StateNotifier<int> {
  CounterNotifier() : super(0);
  void increment() => state = state + 1;
}

// 2. Wrap the app in a ProviderScope (instead of ChangeNotifierProvider)
void main() {
  runApp(const ProviderScope(child: MyApp()));
}

// 3. Consume with a ConsumerWidget instead of StatelessWidget
class CounterDisplay extends ConsumerWidget {
  const CounterDisplay({super.key});
  @override
  Widget build(BuildContext context, WidgetRef ref) {
    final count = ref.watch(counterProvider);
    return Text('Count: $count');
  }
}

class IncrementButton extends ConsumerWidget {
  const IncrementButton({super.key});
  @override
  Widget build(BuildContext context, WidgetRef ref) {
    return ElevatedButton(
      onPressed: () => ref.read(counterProvider.notifier).increment(),
      child: const Text('+1'),
    );
  }
}

Bloc — Strict, Event-Driven State Management

Bloc formalizes state changes as a stream of events in, states out — every state change must be traceable to an explicit event, which makes very large apps with many contributors easier to reason about and test, at the cost of more upfront boilerplate.

// Events — what can happen
abstract class CounterEvent {}
class IncrementPressed extends CounterEvent {}

// The Bloc — maps events to new states
class CounterBloc extends Bloc<CounterEvent, int> {
  CounterBloc() : super(0) {
    on<IncrementPressed>((event, emit) => emit(state + 1));
  }
}

// Providing it, using flutter_bloc's BlocProvider
BlocProvider(
  create: (context) => CounterBloc(),
  child: const CounterScreen(),
)

// Consuming it with BlocBuilder
BlocBuilder<CounterBloc, int>(
  builder: (context, count) => Text('Count: $count'),
)

// Firing an event
context.read<CounterBloc>().add(IncrementPressed());

Choosing Between Them

ApproachGood fit forTrade-off
setStateSmall, single-widget-local stateDoesn't scale to sharing state across screens
ProviderSmall-to-medium apps; official recommendation for most projectsRuntime type lookups; easy to misuse watch vs read
RiverpodSame range as Provider, plus apps wanting compile-time safety and testability outside widgetsSlightly steeper learning curve than Provider
Bloc / flutter_blocLarge teams/apps wanting a strict, event-driven, highly testable architectureMore boilerplate per feature (events + states + bloc)
GetXRapid development with state management, routing, and DI combinedLess idiomatic Flutter; more "magic" than the alternatives above
KEY POINTS
  • State management answers one question: how does a UI change get communicated to everything that depends on it?
  • ChangeNotifier + notifyListeners() + Provider is the most common beginner-to-intermediate pattern in real Flutter apps.
  • context.watch<T>() rebuilds on change; context.read<T>() reads once without subscribing — using the wrong one is a very common source of bugs.

Unit 27Networking & REST APIs

The http package (published by the Dart team) is the standard, lightweight way to make network requests. Add it to pubspec.yaml (http: ^1.2.0) and run flutter pub get before using it.

Making a GET Request

import 'dart:convert';
import 'package:http/http.dart' as http;

Future<List<dynamic>> fetchStudents() async {
  final response = await http.get(
    Uri.parse('https://api.example.com/students'),
  );

  if (response.statusCode == 200) {
    return jsonDecode(response.body) as List<dynamic>;
  } else {
    throw Exception('Failed to load students: ${response.statusCode}');
  }
}

Modeling JSON with a Class

Rather than passing raw Map<String, dynamic> objects around, define a model class with a fromJson factory constructor — this keeps parsing logic in one place and gives you compile-time type safety everywhere else.

class Student {
  final String name;
  final int rollNumber;

  Student({required this.name, required this.rollNumber});

  factory Student.fromJson(Map<String, dynamic> json) {
    return Student(
      name: json['name'] as String,
      rollNumber: json['rollNumber'] as int,
    );
  }
}

Future<List<Student>> fetchStudents() async {
  final response = await http.get(Uri.parse('https://api.example.com/students'));
  final List<dynamic> jsonList = jsonDecode(response.body);
  return jsonList.map((json) => Student.fromJson(json)).toList();
}

Displaying Async Data with FutureBuilder

FutureBuilder<List<Student>>(
  future: fetchStudents(),
  builder: (context, snapshot) {
    if (snapshot.connectionState == ConnectionState.waiting) {
      return const CircularProgressIndicator();
    } else if (snapshot.hasError) {
      return Text('Error: ${snapshot.error}');
    } else {
      final students = snapshot.data!;
      return ListView.builder(
        itemCount: students.length,
        itemBuilder: (context, i) => ListTile(title: Text(students[i].name)),
      );
    }
  },
)
TIPFutureBuilder rebuilds automatically as its future moves through waiting → data/error states — you never need to manually call setState() just to show a loading spinner.

POST Request Example

Future<void> createStudent(String name, int roll) async {
  final response = await http.post(
    Uri.parse('https://api.example.com/students'),
    headers: {'Content-Type': 'application/json'},
    body: jsonEncode({'name': name, 'rollNumber': roll}),
  );
  if (response.statusCode != 201) {
    throw Exception('Failed to create student');
  }
}
KEY POINTS
  • The http package handles GET/POST/PUT/DELETE requests; dart:convert's jsonDecode/jsonEncode handle JSON parsing.
  • A fromJson factory constructor on a model class centralizes parsing and gives the rest of your app type safety.
  • FutureBuilder is the idiomatic Flutter widget for rendering loading/data/error states from a single Future.

Unit 28Local Data Persistence

shared_preferences — simple key-value storage

Ideal for small amounts of data: user settings, a login flag, a theme preference. Not suitable for structured or large datasets.

import 'package:shared_preferences/shared_preferences.dart';

Future<void> saveUsername(String name) async {
  final prefs = await SharedPreferences.getInstance();
  await prefs.setString('username', name);
}

Future<String?> loadUsername() async {
  final prefs = await SharedPreferences.getInstance();
  return prefs.getString('username');
}

sqflite — local SQL database

For structured, relational, and larger datasets stored entirely on-device, sqflite gives you a full local SQLite database.

import 'package:sqflite/sqflite.dart';
import 'package:path/path.dart';

Future<Database> openAppDatabase() async {
  final path = join(await getDatabasesPath(), 'app.db');
  return openDatabase(
    path,
    version: 1,
    onCreate: (db, version) {

Unit 29Animations

Implicit Animations — the easy 80%

Widgets prefixed Animated* handle the entire animation for you — just change a property, and the widget smoothly animates from the old value to the new one.

class FadeBox extends StatefulWidget {
  const FadeBox({super.key});
  @override
  State<FadeBox> createState() => _FadeBoxState();
}

class _FadeBoxState extends State<FadeBox> {
  double _opacity = 1.0;

  @override
  Widget build(BuildContext context) {
    return GestureDetector(
      onTap: () => setState(() => _opacity = _opacity == 1.0 ? 0.2 : 1.0),
      child: AnimatedOpacity(

        duration: const Duration(milliseconds: 400),
        child: Container(width: 100, height: 100, color: Colors.teal),
      ),
    );
  }
}

Explicit Animations — full control

When you need precise control (looping, reversing, chaining multiple properties together), use an AnimationController directly. It requires a TickerProvider, which is why the State class mixes in SingleTickerProviderStateMixin.

class PulsingIcon extends StatefulWidget {
  const PulsingIcon({super.key});
  @override
  State<PulsingIcon> createState() => _PulsingIconState();
}

class _PulsingIconState extends State<PulsingIcon>
    with SingleTickerProviderStateMixin {
  late AnimationController _controller;
  late Animation<double> _scale;

  @override
  void initState() {
    super.initState();
    _controller = AnimationController(
      vsync: this,
      duration: const Duration(seconds: 1),
    )..repeat(reverse: true);
    _scale = Tween<double>(begin: 0.8, end: 1.2).animate(_controller);
  }

  @override
  void dispose() {
    _controller.dispose();
    super.dispose();
  }

  @override
  Widget build(BuildContext context) {
    return ScaleTransition(scale: _scale, child: const Icon(Icons.favorite, size: 48));
  }
}
CAUTIONEvery AnimationController you create must be disposed in dispose(). A forgotten controller keeps ticking in the background after its widget is gone, wasting resources and, in some cases, causing errors when it eventually calls setState() on an unmounted widget.
KEY POINTS
  • Animated* widgets cover simple property-change animations with almost no boilerplate.
  • AnimationController + Tween gives full control for loops, reversals, and custom curves.
  • Always dispose an AnimationController in the State's dispose() method.

Unit 34Custom Widgets & Composition

Flutter's guiding philosophy is composition over configuration — instead of one giant, deeply-configurable widget, you build small, focused widgets and combine them. Extracting a repeated chunk of UI into its own widget class is the single most impactful habit for keeping a Flutter codebase readable as it grows.

Extracting a Reusable Widget

class StatCard extends StatelessWidget {
  final String label;
  final String value;
  final IconData icon;

  const StatCard({
    super.key,
    required this.label,
    required this.value,
    required this.icon,
  });

  @override
  Widget build(BuildContext context) {
    return Card(
      child: Padding(
        padding: const EdgeInsets.all(16),
        child: Row(
          children: [
            Icon(icon, size: 32, color: Theme.of(context).primaryColor),
            const SizedBox(width: 12),
            Column(
              crossAxisAlignment: CrossAxisAlignment.start,
              children: [
                Text(value, style: const TextStyle(fontSize: 20, fontWeight: FontWeight.bold)),
                Text(label, style: const TextStyle(color: Colors.grey)),
              ],
            ),
          ],
        ),
      ),
    );
  }
}

// Reused anywhere, with different data:
const StatCard(label: 'Papers Published', value: '118', icon: Icons.article)
TIPA common rule of thumb: if a build() method exceeds roughly 60-80 lines, or the same widget subtree appears more than once, it's usually time to extract a new widget class rather than a helper method that merely returns a Widget — real widget classes give Flutter's diffing algorithm useful information that helper methods don't.

Composing with child and children

Accepting a Widget child (or List<Widget> children) parameter, rather than hard-coding content, is what makes your own widgets as flexible as Flutter's built-in ones.

class RoundedPanel extends StatelessWidget {
  final Widget child;
  const RoundedPanel({super.key, required this.child});

  @override
  Widget build(BuildContext context) {
    return Container(
      padding: const EdgeInsets.all(16),
      decoration: BoxDecoration(
        color: Colors.white,
        borderRadius: BorderRadius.circular(12),
        boxShadow: [BoxShadow(color: Colors.black12, blurRadius: 8)],
      ),
      child: child,
    );
  }
}
KEY POINTS
  • Extract a widget class, not just a method, whenever UI is reused or a build() method grows large.
  • Accepting child/children parameters is what makes a custom widget composable rather than a rigid, one-off template.
  • Composition is the core Flutter design pattern — nearly every built-in widget is itself built by composing smaller ones.

Unit 35Slivers & CustomScrollView

A sliver is a portion of a scrollable area that Flutter can lay out, paint, and clip lazily — the low-level mechanism behind every scrolling widget you've already used, including ListView. Working with slivers directly becomes necessary once a screen needs to mix several *different kinds* of scrolling content in one continuous scroll — for example, a collapsing header image above a list.

CustomScrollView with SliverAppBar

CustomScrollView(
  slivers: [
    SliverAppBar(
      expandedHeight: 200,
      pinned: true,
      flexibleSpace: FlexibleSpaceBar(
        title: const Text('AI & DS Department'),
        background: Image.asset('assets/images/banner.png', fit: BoxFit.cover),
      ),
    ),
    SliverList(
      delegate: SliverChildBuilderDelegate(
        (context, index) => ListTile(title: Text('Item $index')),
        childCount: 20,
      ),
    ),
    SliverGrid(
      gridDelegate: const SliverGridDelegateWithFixedCrossAxisCount(crossAxisCount: 2),
      delegate: SliverChildBuilderDelegate(
        (context, index) => Card(child: Center(child: Text('Grid $index'))),
        childCount: 6,

  ],

TIPAs soon as you need a scroll effect that a plain ListView or Column can't express — a header that shrinks as you scroll, mixed list-and-grid sections, sticky section titles — reach for CustomScrollView and its Sliver* children instead of nesting scrollables inside each other, which Flutter does not support well.
KEY POINTS
  • Slivers are the lazy-layout primitive underneath every Flutter scrolling widget.
  • CustomScrollView lets you combine SliverAppBar, SliverList, and SliverGrid into one continuous, coordinated scroll.
  • Reach for slivers specifically when a screen needs scroll effects beyond what ListView/Column alone can express.

Unit 36Custom Painting with CustomPainter

For UI that isn't made of standard shapes — custom charts, signature pads, progress rings, hand-drawn diagrams — Flutter exposes the same low-level Canvas API its own widgets are built on, via CustomPainter.

class CirclePainter extends CustomPainter {
  final double progress; // 0.0 to 1.0
  CirclePainter(this.progress);

  @override
  void paint(Canvas canvas, Size size) {
    final center = Offset(size.width / 2, size.height / 2);
    final radius = size.width / 2;

    final backgroundPaint = Paint()
      ..color = Colors.grey.shade300
      ..style = PaintingStyle.stroke
      ..strokeWidth = 8;
    canvas.drawCircle(center, radius, backgroundPaint);

    final progressPaint = Paint()
      ..color = Colors.indigo
      ..style = PaintingStyle.stroke
      ..strokeWidth = 8
      ..strokeCap = StrokeCap.round;
    canvas.drawArc(
      Rect.fromCircle(center: center, radius: radius),
      -3.14159 / 2,           // start at the top
      2 * 3.14159 * progress, // sweep angle based on progress
      false,
      progressPaint,
    );
  }

  @override
  bool shouldRepaint(CirclePainter oldDelegate) => oldDelegate.progress != progress;
}

// Usage:
CustomPaint(size: const Size(100, 100), painter: CirclePainter(0.7))
TIPAlways implement shouldRepaint deliberately — returning true unconditionally forces a repaint on every frame regardless of whether anything actually changed, which is wasteful for anything animated or frequently rebuilt.
KEY POINTS
  • CustomPainter gives direct Canvas access for shapes, charts, and effects no built-in widget provides.
  • shouldRepaint controls whether an expensive repaint actually happens — return false when the painted output hasn't changed.
  • Wrap a CustomPainter in the CustomPaint widget, giving it an explicit size when it has no child to size itself by.

Unit 37Responsive & Adaptive Design

A Flutter app might run on a phone, a tablet, a foldable, or a desktop window of any size — good layouts adapt rather than assume a fixed screen size. Two tools cover the great majority of real-world responsive needs.

MediaQuery — read the available space

Widget build(BuildContext context) {
  final width = MediaQuery.of(context).size.width;
  final isTablet = width > 600;

  return isTablet
      ? const TwoColumnLayout()
      : const SingleColumnLayout();
}

LayoutBuilder — react to the parent's constraints

MediaQuery reports the whole screen's size; LayoutBuilder reports the space actually available to *this* widget from its parent — the more correct choice when building a reusable widget that might be placed inside a sidebar, a dialog, or a full screen.

LayoutBuilder(
  builder: (context, constraints) {
    if (constraints.maxWidth > 600) {
      return const GridView.count(crossAxisCount: 3, children: []);
    } else {
      return const GridView.count(crossAxisCount: 1, children: []);
    }
  },
)
ToolReportsUse when
MediaQuery.of(context)The full screen/window size, orientation, padding (e.g. notches)Deciding overall app layout, text scaling, safe areas
LayoutBuilderThe constraints handed down from this widget's immediate parentBuilding a self-contained, reusable widget that adapts regardless of where it's placed
KEY POINTS
  • MediaQuery gives screen-level information; LayoutBuilder gives parent-level constraints — use the one that matches the actual question you're asking.
  • A common breakpoint convention: under ~600 logical pixels wide = phone layout, 600–900 = tablet, above 900 = desktop/large tablet — adjust to your app's actual content.
  • Design layouts to respond to available space rather than hard-coding pixel values, so the same codebase looks correct across phones, tablets, and desktop windows.

Unit 38Accessibility

Flutter builds a parallel semantics tree alongside the widget tree specifically for screen readers (TalkBack on Android, VoiceOver on iOS) and other assistive technology. Most built-in widgets (buttons, text fields, icons with a tooltip) already populate reasonable semantics automatically; the Semantics widget lets you fill in the gaps for custom widgets.

Semantics(
  label: 'Submit quarterly report',
  button: true,
  child: GestureDetector(
    onTap: submitReport,
    child: Container(
      padding: const EdgeInsets.all(12),
      color: Colors.indigo,
      child: const Icon(Icons.send, color: Colors.white),
    ),
  ),
)

A Practical Accessibility Checklist

  • Every tappable Icon-only button has either a tooltip or an explicit Semantics(label: ...).
  • Text and background colors meet a reasonable contrast ratio — don't rely on color alone to convey meaning (e.g. red/green status).
  • Tap targets are at least 48×48 logical pixels, per Material Design guidance.
  • Text respects the user's device font-scale setting rather than being locked to a fixed pixel size wherever reasonable.
  • Test the app with TalkBack/VoiceOver turned on at least once before release — issues that are invisible visually are often immediately obvious by ear.
KEY POINTS
  • Flutter maintains a semantics tree specifically for screen readers; most built-in widgets populate it automatically.
  • Use the Semantics widget to describe custom, non-standard widgets that lack an inherent accessible meaning.
  • Accessibility is a checklist worth reviewing before every release, not a one-time setup step.

Unit 40Platform Channels — Talking to Native Code

When a feature genuinely isn't available through any Flutter package — a proprietary native SDK, a very device-specific API — platform channels let Dart code call into native Android (Kotlin/Java) or iOS (Swift/Objective-C) code, and receive a result back, asynchronously.

Dart Side

import 'package:flutter/services.dart';

const platform = MethodChannel('com.lbrce.aids/battery');

Future<int> getBatteryLevel() async {
  try {
    final int result = await platform.invokeMethod('getBatteryLevel');
    return result;
  } on PlatformException catch (e) {
    print('Failed to get battery level: ${e.message}');
    return -1;
  }
}

Android Side (Kotlin, simplified)

// android/app/src/main/kotlin/.../MainActivity.kt
MethodChannel(flutterEngine.dartExecutor.binaryMessenger, "com.lbrce.aids/battery")
    .setMethodCallHandler { call, result ->
        if (call.method == "getBatteryLevel") {
            val level = getBatteryLevelNative()
            result.success(level)
        } else {
            result.notImplemented()
        }
    }
TIPBefore reaching for a custom platform channel, search pub.dev thoroughly — the vast majority of native functionality (battery, sensors, camera, biometrics, connectivity) already has a well-maintained plugin wrapping the platform channel for you.
KEY POINTS
  • Platform channels are the bridge between Dart and native Android/iOS code, used only when no existing plugin covers the need.
  • MethodChannel sends a named method call with arguments and awaits a single result — conceptually similar to a Future-based RPC.
  • Most plugins on pub.dev (camera, sensors, notifications, etc.) are themselves built on top of platform channels.

  • Build flavors let dev/staging/production variants of an app coexist and point at different backends without maintaining separate codebases.

Unit 44App Architecture & Dependency Injection

As a Flutter app grows past a handful of screens, mixing UI, business logic, and data-fetching directly inside widgets becomes hard to test and maintain. A layered architecture separates these concerns; dependency injection (DI) is the technique of supplying a class's dependencies from the outside rather than having it construct them itself, which is what makes each layer independently testable.

A Simple Repository Pattern

// 1. Abstract contract — the UI and business logic depend on this, not a concrete implementation
abstract class StudentRepository {
  Future<List<Student>> getStudents();
}

// 2. A concrete implementation talking to a real API
class ApiStudentRepository implements StudentRepository {
  final http.Client client;
  ApiStudentRepository(this.client);

  @override
  Future<List<Student>> getStudents() async {
    final response = await client.get(Uri.parse('https://api.example.com/students'));
    final list = jsonDecode(response.body) as List;
    return list.map((j) => Student.fromJson(j)).toList();
  }
}

// 3. A fake implementation — trivially swapped in for tests, no network needed
class FakeStudentRepository implements StudentRepository {
  @override
  Future<List<Student>> getStudents() async =>
      [Student(name: 'Test Student', rollNumber: 1)];
}

Injecting the Dependency

class StudentListScreen extends StatelessWidget {
  final StudentRepository repository; // supplied from outside, not created here
  const StudentListScreen({super.key, required this.repository});

  @override
  Widget build(BuildContext context) {
    return FutureBuilder<List<Student>>(
      future: repository.getStudents(),
      builder: (context, snapshot) {
        // ... render based on snapshot, exactly as in Unit 21
        return const SizedBox();
      },
    );
  }
}

// Production: StudentListScreen(repository: ApiStudentRepository(http.Client()))
// Testing:    StudentListScreen(repository: FakeStudentRepository())
TIPFor larger apps, packages like get_it (a simple service locator) or riverpod (which has DI built into its provider model) automate wiring dependencies together, but the underlying idea — depend on an abstraction, supply the concrete implementation from outside — stays the same.
LayerResponsibility
Presentation (widgets)Display data and forward user actions — no business logic or direct API/database calls
Domain / business logicApplication rules, validation, orchestration — framework-agnostic where possible
Data (repositories)Fetching and persisting data — the only layer that knows about the API, database, or cache
KEY POINTS
  • Dependency injection means a class receives its dependencies from outside rather than constructing them itself.
  • Coding against an abstract repository interface, not a concrete implementation, is what makes swapping in a fake/test version trivial.
  • Separating presentation, domain, and data layers keeps a growing app testable and easier to reason about.

Unit 45Integration Testing

Integration tests (via the integration_test package) run the complete, real app — not a simulated widget environment — on an actual device or emulator, driving it exactly the way a user would. They're slower than unit or widget tests but catch problems those levels cannot, such as issues specific to a real platform's rendering or plugin behaviour.

# pubspec.yaml
dev_dependencies:
  integration_test:
    sdk: flutter
// integration_test/app_test.dart
import 'package:flutter_test/flutter_test.dart';
import 'package:integration_test/integration_test.dart';
import 'package:my_app/main.dart' as app;

void main() {
  IntegrationTestWidgetsFlutterBinding.ensureInitialized();

  testWidgets('full login flow', (tester) async {
    app.main();
    await tester.pumpAndSettle();

    await tester.enterText(find.byKey(const Key('emailField')), 'test@lbrce.ac.in');
    await tester.enterText(find.byKey(const Key('passwordField')), 'password123');
    await tester.tap(find.byKey(const Key('loginButton')));
    await tester.pumpAndSettle();

    expect(find.text('Welcome'), findsOneWidget);
  });
}
flutter test integration_test/app_test.dart
TIPpumpAndSettle() repeatedly pumps frames until no more are scheduled — ideal after actions that trigger animations or async work, so the test waits for the UI to fully finish reacting before making assertions.
KEY POINTS
  • Integration tests exercise the real, compiled app on a device/emulator — the closest automated testing gets to actual user behaviour.
  • find.byKey(...) is the most robust way to locate widgets in tests, since it doesn't depend on visible text that might change or be localized.
  • pumpAndSettle() waits out animations and async work before assertions run.

Unit 46Performance Optimization & Best Practices

Flutter targets 60 (or 120, on capable devices) frames per second by default. Most performance problems come from rebuilding, laying out, or repainting more of the widget tree than actually changed.

Use const Constructors Wherever Possible

// Rebuilt every time the parent rebuilds, even though nothing about it changes
Text('Static Label')

// Built once and reused — Flutter skips rebuilding it entirely
const Text('Static Label')

Keep build() Methods Cheap

  • Never perform network calls, heavy computation, or file I/O directly inside build() — it can run many times per second during animations.
  • Extract widgets that don't depend on frequently-changing state into their own const or separately-built classes, rather than rebuilding them as part of a large parent's build().
  • Prefer ListView.builder/GridView.builder over eagerly building every item up front for any list that could grow large (Unit 17).

Scope Rebuilds Precisely

// Rebuilds the ENTIRE subtree whenever CounterModel changes
Consumer<CounterModel>(
  builder: (context, model, child) => Column(
    children: [
      ExpensiveStaticHeader(), // gets needlessly rebuilt too
      Text('${model.count}'),
    ],
  ),
)

// Better — the 'child' parameter is built once and passed through untouched
Consumer<CounterModel>(
  builder: (context, model, child) => Column(
    children: [child!, Text('${model.count}')],
  ),
  child: const ExpensiveStaticHeader(),
)

Diagnose, Don't Guess

ToolShows you
DevTools Performance tabFrame render times; spot janky frames over 16ms (60fps) or 8ms (120fps)
"Repaint Rainbow" (debugRepaintRainbowEnabled)Visually flashes any region that repaints, revealing unnecessary repaints at a glance
Widget rebuild counts (DevTools)Which widgets rebuild far more often than expected
KEY POINTS
  • const constructors let Flutter skip rebuilding widgets that provably haven't changed — use them by default wherever a widget's arguments are compile-time constants.
  • Keep build() fast and side-effect-free; heavy work belongs in a repository/service layer, not inline in the widget tree.
  • The child parameter pattern (seen in AnimatedBuilder and Consumer) lets you exclude a static subtree from rebuilding.
  • Use DevTools to measure before optimizing — intuition about what's "probably slow" in Flutter is often wrong.

Unit 47App Lifecycle & State Restoration

A mobile app moves through several lifecycle states as the user switches apps, locks the phone, or the OS reclaims memory. Reacting correctly — pausing a video, saving a draft, disconnecting a stream — prevents wasted resources and, at worst, data loss.

Observing Lifecycle Changes

class LifecycleAwareScreen extends StatefulWidget {
  const LifecycleAwareScreen({super.key});
  @override
  State<LifecycleAwareScreen> createState() => _LifecycleAwareScreenState();
}

class _LifecycleAwareScreenState extends State<LifecycleAwareScreen>
    with WidgetsBindingObserver {
  @override
  void initState() {
    super.initState();
    WidgetsBinding.instance.addObserver(this);
  }

  @override
  void dispose() {
    WidgetsBinding.instance.removeObserver(this);
    super.dispose();
  }

  @override
  void didChangeAppLifecycleState(AppLifecycleState state) {
    if (state == AppLifecycleState.paused) {
      print('App moved to background — save draft state here');
    } else if (state == AppLifecycleState.resumed) {
      print('App back in foreground — refresh data if needed');
    }
  }

  @override
  Widget build(BuildContext context) => const SizedBox();
}
StateMeaning
resumedApp is visible and responding to user input
inactiveApp is in the foreground but not receiving input (e.g. an incoming call overlay)
pausedApp is not visible — moved to the background
detachedApp is still hosted by Flutter but detached from any view (rare, mostly during teardown)
KEY POINTS
  • WidgetsBindingObserver + didChangeAppLifecycleState is how a widget reacts to the app moving to background/foreground.
  • paused is the natural place to persist unsaved work; resumed is a natural place to refresh stale data.
  • Always remove the observer in dispose(), mirroring the pairing rule from other lifecycle-bound resources (Unit 15, Unit 23).

Unit 48Error Handling & Crash Reporting in Production

A shipped app needs a strategy for errors beyond a local try/catch — both to avoid a blank, crashed screen for the user, and to actually find out an error happened at all once the app is in someone else's hands.

Catching Flutter Framework Errors

void main() {
  FlutterError.onError = (FlutterErrorDetails details) {
    // Log to console during development
    FlutterError.presentError(details);
    // In production, forward to a crash-reporting service instead/also:
    // FirebaseCrashlytics.instance.recordFlutterFatalError(details);
  };
  runApp(const MyApp());
}

Catching Everything Else with runZonedGuarded

FlutterError.onError only covers errors thrown during widget building/layout/painting. Errors in your own async code (a background Future, an uncaught exception in a callback) need a Zone-level guard to be caught centrally.

import 'dart:async';

void main() {
  runZonedGuarded(() {
    runApp(const MyApp());
  }, (error, stackTrace) {
    print('Uncaught error: $error');
    // FirebaseCrashlytics.instance.recordError(error, stackTrace);
  });
}

A User-Friendly Fallback UI

ErrorWidget.builder = (FlutterErrorDetails details) {
  return Scaffold(
    body: Center(
      child: Text('Something went wrong. Please restart the app.'),
    ),
  );
};
TIPfirebase_crashlytics and sentry_flutter are the two most widely used crash-reporting services in the Flutter ecosystem — both provide drop-in hooks for exactly the two handlers shown above, plus a dashboard of every crash, its stack trace, and how many users it affected.
KEY POINTS
  • FlutterError.onError catches framework-level errors (build/layout/paint); runZonedGuarded catches everything else, including async errors.
  • ErrorWidget.builder replaces Flutter's default red error screen with something presentable to end users.
  • A crash-reporting service (Crashlytics, Sentry) is what turns "a user's app is crashing somewhere" into an actionable, specific bug report.

Unit 49Working with Camera, Gallery & Media

The image_picker package covers the overwhelming majority of apps' media needs — capturing a photo, recording video, or picking existing media from the gallery — through one simple, cross-platform API.

import 'package:image_picker/image_picker.dart';

final ImagePicker _picker = ImagePicker();

Future<void> pickFromGallery() async {
  final XFile? image = await _picker.pickImage(source: ImageSource.gallery);
  if (image != null) {
    print('Picked: ${image.path}');
  }
}

Future<void> takePhoto() async {
  final XFile? photo = await _picker.pickImage(
    source: ImageSource.camera,
    maxWidth: 1200,
    imageQuality: 85,
  );
  if (photo != null) {
    print('Captured: ${photo.path}');
  }
}

Displaying the Picked Image

import 'dart:io';

Image.file(File(image.path))
CAUTIONCamera and gallery access require platform-specific permission declarations — NSCameraUsageDescription/NSPhotoLibraryUsageDescription in iOS's Info.plist, and the CAMERA permission in Android's manifest — plus a runtime permission request via a package like permission_handler on Android 6.0+ and iOS.
KEY POINTS
  • image_picker handles both camera capture and gallery selection through one unified API returning an XFile.
  • maxWidth/imageQuality let you downscale/compress an image at pick time, avoiding unnecessarily large uploads later.

Unit 50Geolocation & Maps Overview

The geolocator package is the standard way to read a device's GPS location; google_maps_flutter (or flutter_map for an open-source, non-Google alternative) renders an interactive map.

Getting the Current Location

import 'package:geolocator/geolocator.dart';

Future<Position> getCurrentLocation() async {
  LocationPermission permission = await Geolocator.checkPermission();
  if (permission == LocationPermission.denied) {
    permission = await Geolocator.requestPermission();
  }

  return Geolocator.getCurrentPosition(
    desiredAccuracy: LocationAccuracy.high,
  );
}

Displaying a Map

import 'package:google_maps_flutter/google_maps_flutter.dart';

GoogleMap(

  },
)
TIPgoogle_maps_flutter requires an API key configured natively for both Android (AndroidManifest.xml) and iOS (AppDelegate.swift) — this is a native platform setup step, not something Dart code alone can configure.
KEY POINTS
  • Always check and, if needed, request location permission before calling getCurrentPosition() — it throws if permission was never granted.
  • google_maps_flutter needs a valid, natively-configured Google Maps API key per platform before it will render anything.
  • flutter_map is a common Google-Maps-free alternative when avoiding Google API keys/billing is a requirement.

Unit 51WebSockets & Real-Time Communication

REST APIs (Unit 21) are request/response — the client always initiates. For genuinely real-time features (live chat, live scores, collaborative editing), a WebSocket keeps a persistent, two-way connection open, letting the server push data the instant it changes.

import 'dart:convert';
import 'package:web_socket_channel/web_socket_channel.dart';

class ChatService {
  final channel = WebSocketChannel.connect(
    Uri.parse('wss://example.com/chat'),
  );

  Stream<dynamic> get messages => channel.stream;

  void sendMessage(String text) {
    channel.sink.add(jsonEncode({'message': text}));
  }

  void close() {
    channel.sink.close();
  }
}
// In a widget — pair the Stream directly with StreamBuilder (Unit 21)
StreamBuilder(
  stream: chatService.messages,
  builder: (context, snapshot) {
    if (!snapshot.hasData) return const Text('Waiting for messages...');
    return Text('New message: ${snapshot.data}');
  },
)
CAUTIONAlways close a WebSocket channel (channel.sink.close()) when the screen using it is disposed — an open socket left dangling keeps consuming battery and network resources in the background.
KEY POINTS
  • WebSockets keep a persistent connection open, allowing the server to push updates without the client polling repeatedly.
  • web_socket_channel exposes an incoming Stream and an outgoing sink — the Stream half pairs naturally with StreamBuilder.
  • Always close the channel when it's no longer needed, mirroring the disposal discipline used for controllers and animations.

Unit 52Advanced Forms & Multi-Step Wizards

Beyond single-screen forms (Unit 18), many real apps need multi-field cross-validation or a multi-step "wizard" flow — a registration process broken across several screens that still validates and submits as one logical unit.

Cross-Field Validation

final _passwordController = TextEditingController();

TextFormField(
  decoration: const InputDecoration(labelText: 'Confirm Password'),
  validator: (value) {
    if (value != _passwordController.text) {
      return 'Passwords do not match';
    }
    return null;
  },
)

A Simple Multi-Step Wizard with Stepper

int _currentStep = 0;

Stepper(
  currentStep: _currentStep,
  onStepContinue: () {
    if (_currentStep < 2) {
      setState(() => _currentStep++);
    } else {
      print('Wizard complete — submit all collected data');
    }
  },
  onStepCancel: () {
    if (_currentStep > 0) setState(() => _currentStep--);
  },
  steps: const [
    Step(title: Text('Personal Details'), content: Text('Name, email...')),
    Step(title: Text('Academic Details'), content: Text('Roll number, year...')),
    Step(title: Text('Review'), content: Text('Confirm and submit')),
  ],
)
TIPFor a wizard spanning multiple full screens (rather than one scrollable Stepper), hold the shared, accumulated form data in a single object owned above all the steps — via Provider/Riverpod (Unit 20) — rather than trying to pass and merge partial data manually through Navigator route arguments.
KEY POINTS
  • Cross-field validators compare against another field's current value, typically read from its TextEditingController.
  • Stepper provides a ready-made, linear multi-step UI for forms that fit comfortably on one screen.
  • For wizards spanning multiple full screens, centralize the shared form state above the steps rather than threading it through navigation arguments.
