PART III — CAPSTONE PROJECT
Building a complete app end-to-end, integrating everything covered so far

Step 1Project Overview & Requirements

This capstone walks through building a small, complete, working app — a Faculty Activity Tracker — deliberately chosen because it mirrors real departmental workflows: recording events, tracking student participation, and reviewing a running history, all backed by local persistence. Every concept from Parts I–III is used somewhere in this build: widgets and layout, forms, navigation, state management, local storage, and testing.

Requirements

  • A home screen listing all recorded activities, newest first.
  • An "Add Activity" screen with a validated form (title, category, date, description).
  • Data persists locally, so the list survives an app restart.
  • A details screen for viewing (and deleting) a single activity.
  • A running count of total activities, always visible, updating live as items are added or removed.

Planned Architecture

LayerResponsibilityBuilt with
ModelDefines what an Activity isPlain Dart class
DataReads/writes activities to disksqflite repository
StateHolds the current list in memory, notifies the UI on changeChangeNotifier + Provider
UIHome list, add form, details screenStatelessWidget/StatefulWidget, Navigator
TIPNotice this is exactly the layered architecture from Unit 24 (App Architecture & Dependency Injection) — a small app benefits from the same separation of concerns as a large one, just with fewer layers actually implemented.
KEY POINTS
  • A clear, small requirements list before writing any code keeps a capstone project from sprawling.
  • Even a small app benefits from separating model, data, state, and UI concerns from the start.

Step 2The Data Model

Following the pattern from Unit 6 (OOP) and Unit 21 (JSON modeling), the Activity class is a plain, immutable data holder with a factory constructor for reading from the database and a toMap() method for writing to it.

class Activity {
  final int? id; // null until saved — the database assigns the real id
  final String title;
  final String category;
  final DateTime date;
  final String description;

  const Activity({
    this.id,
    required this.title,
    required this.category,
    required this.date,
    required this.description,
  });

  Map<String, dynamic> toMap() {
    return {
      'id': id,
      'title': title,
      'category': category,
      'date': date.toIso8601String(),
      'description': description,
    };
  }

  factory Activity.fromMap(Map<String, dynamic> map) {
    return Activity(
      id: map['id'] as int,
      title: map['title'] as String,
      category: map['category'] as String,
      date: DateTime.parse(map['date'] as String),
      description: map['description'] as String,
    );
  }

  // Copy with a real id once the database assigns one
  Activity copyWith({int? id}) {
    return Activity(
      id: id ?? this.id,
      title: title,
      category: category,
      date: date,
      description: description,
    );
  }
}
TIPThe copyWith pattern — creating a modified clone of an otherwise-immutable object — is extremely common in Flutter codebases, precisely because keeping model classes immutable (all final fields) avoids an entire category of accidental-mutation bugs.
KEY POINTS
  • toMap()/fromMap() mirror toJson()/fromJson() from Unit 21, just targeting a local database row instead of a network response.
  • Keeping the model immutable (final fields, no setters) and using copyWith for changes avoids accidental, hard-to-trace mutation bugs.

Step 3The Data Layer (Repository + sqflite)

Following the repository pattern from Unit 24, all database code is isolated in one class — nothing else in the app talks to sqflite directly.

import 'package:sqflite/sqflite.dart';
import 'package:path/path.dart';

class ActivityRepository {
  Database? _db;

  Future<Database> get database async {
    if (_db != null) return _db!;
    _db = await _initDb();
    return _db!;
  }

  Future<Database> _initDb() async {
    final path = join(await getDatabasesPath(), 'activities.db');
    return openDatabase(
      path,
      version: 1,
      onCreate: (db, version) => db.execute('''
        CREATE TABLE activities(
          id INTEGER PRIMARY KEY AUTOINCREMENT,
          title TEXT NOT NULL,
          category TEXT NOT NULL,
          date TEXT NOT NULL,
          description TEXT
        )
      '''),
    );
  }

  Future<List<Activity>> getAll() async {
    final db = await database;
    final rows = await db.query('activities', orderBy: 'date DESC');
    return rows.map((row) => Activity.fromMap(row)).toList();
  }

  Future<Activity> insert(Activity activity) async {
    final db = await database;
    final id = await db.insert('activities', activity.toMap());
    return activity.copyWith(id: id);
  }

  Future<void> delete(int id) async {
    final db = await database;
    await db.delete('activities', where: 'id = ?', whereArgs: [id]);
  }
}
CAUTIONNotice where: 'id = ?', whereArgs: [id] rather than building the SQL string with interpolation directly. Parameterized queries are the standard defense against SQL injection and malformed queries — a habit worth building even for a local, single-user database.
KEY POINTS
  • The repository is the *only* class that imports sqflite — everything else in the app works with plain Activity objects.
  • Lazy-initializing _db (creating the connection only on first use) avoids unnecessary work if the database is never actually touched in a given session.
  • Always use parameterized where/whereArgs rather than building SQL strings by hand.

Step 4The State Layer (ChangeNotifier)

The ActivityListModel is the single source of truth the UI observes, exactly following the Provider pattern from Unit 20. It owns the in-memory list and delegates persistence to the repository.

import 'package:flutter/foundation.dart';

class ActivityListModel extends ChangeNotifier {
  final ActivityRepository _repository;
  List<Activity> _activities = [];
  bool _isLoading = false;

  ActivityListModel(this._repository);

  List<Activity> get activities => _activities;
  bool get isLoading => _isLoading;
  int get totalCount => _activities.length;

  Future<void> load() async {
    _isLoading = true;
    notifyListeners();

    _activities = await _repository.getAll();

    _isLoading = false;
    notifyListeners();
  }

  Future<void> addActivity(Activity activity) async {
    final saved = await _repository.insert(activity);
    _activities = [saved, ..._activities];
    notifyListeners();
  }

  Future<void> deleteActivity(int id) async {
    await _repository.delete(id);
    _activities = _activities.where((a) => a.id != id).toList();
    notifyListeners();
  }
}
KEY POINTS
  • The model exposes read-only getters (activities, isLoading, totalCount) and mutating methods — nothing outside this class ever assigns _activities directly.
  • Every mutating method follows the same shape: update the repository, update local state, call notifyListeners() — this consistency makes the class easy to extend later.

Step 5Wiring the App & the Home Screen

void main() {
  runApp(
    ChangeNotifierProvider(
      create: (context) => ActivityListModel(ActivityRepository())..load(),
      child: const MyApp(),
    ),
  );
}

class MyApp extends StatelessWidget {
  const MyApp({super.key});
  @override
  Widget build(BuildContext context) {
    return MaterialApp(
      title: 'Faculty Activity Tracker',
      theme: ThemeData(colorScheme: ColorScheme.fromSeed(seedColor: Colors.indigo)),
      home: const HomeScreen(),
    );
  }
}
class HomeScreen extends StatelessWidget {
  const HomeScreen({super.key});

  @override
  Widget build(BuildContext context) {
    final model = context.watch<ActivityListModel>();

    return Scaffold(
      appBar: AppBar(
        title: Text('Activities (${model.totalCount})'),
      ),
      body: model.isLoading
          ? const Center(child: CircularProgressIndicator())
          : model.activities.isEmpty
              ? const Center(child: Text('No activities yet. Tap + to add one.'))
              : ListView.builder(
                  itemCount: model.activities.length,
                  itemBuilder: (context, index) {
                    final activity = model.activities[index];
                    return ListTile(
                      title: Text(activity.title),
                      subtitle: Text('${activity.category} · ${activity.date.toLocal()}'.split(' ').first),
                      onTap: () => Navigator.push(
                        context,
                        MaterialPageRoute(
                          builder: (_) => DetailsScreen(activity: activity),
                        ),
                      ),
                    );
                  },
                ),
      floatingActionButton: FloatingActionButton(
        onPressed: () => Navigator.push(
          context,
          MaterialPageRoute(builder: (_) => const AddActivityScreen()),
        ),
        child: const Icon(Icons.add),
      ),
    );
  }
}
TIPThe AppBar title reads model.totalCount directly, and because HomeScreen calls context.watch<ActivityListModel>(), that count updates automatically the instant an activity is added or deleted — no manual refresh logic anywhere.
KEY POINTS
  • ..load() (a cascade) kicks off the initial data load the moment the model is created, without needing a separate initState().
  • context.watch<T>() at the top of build() is what makes the entire screen reactive to the model's changes.

Step 6The Add Activity Form

class AddActivityScreen extends StatefulWidget {
  const AddActivityScreen({super.key});
  @override
  State<AddActivityScreen> createState() => _AddActivityScreenState();
}

class _AddActivityScreenState extends State<AddActivityScreen> {
  final _formKey = GlobalKey<FormState>();
  final _titleController = TextEditingController();
  final _descriptionController = TextEditingController();
  String _category = 'Workshop';
  DateTime _date = DateTime.now();

  @override
  void dispose() {
    _titleController.dispose();
    _descriptionController.dispose();
    super.dispose();
  }

  Future<void> _pickDate() async {
    final picked = await showDatePicker(
      context: context,
      initialDate: _date,
      firstDate: DateTime(2020),
      lastDate: DateTime(2030),
    );
    if (picked != null) setState(() => _date = picked);
  }

  void _submit() {
    if (_formKey.currentState!.validate()) {
      final activity = Activity(
        title: _titleController.text,
        category: _category,
        date: _date,
        description: _descriptionController.text,
      );
      context.read<ActivityListModel>().addActivity(activity);
      Navigator.pop(context);
    }
  }

  @override
  Widget build(BuildContext context) {
    return Scaffold(
      appBar: AppBar(title: const Text('Add Activity')),
      body: Padding(
        padding: const EdgeInsets.all(16),
        child: Form(
          key: _formKey,
          child: Column(
            crossAxisAlignment: CrossAxisAlignment.stretch,
            children: [
              TextFormField(
                controller: _titleController,
                decoration: const InputDecoration(labelText: 'Title'),
                validator: (v) => (v == null || v.isEmpty) ? 'Title is required' : null,
              ),
              const SizedBox(height: 12),
              DropdownButtonFormField<String>(
                value: _category,
                decoration: const InputDecoration(labelText: 'Category'),
                items: ['Workshop', 'Seminar', 'Publication', 'Other']
                    .map((c) => DropdownMenuItem(value: c, child: Text(c)))
                    .toList(),
                onChanged: (v) => setState(() => _category = v!),
              ),
              const SizedBox(height: 12),
              ListTile(
                contentPadding: EdgeInsets.zero,
                title: Text('Date: ${_date.toLocal()}'.split(' ').first),
                trailing: const Icon(Icons.calendar_today),
                onTap: _pickDate,
              ),
              const SizedBox(height: 12),
              TextFormField(
                controller: _descriptionController,
                decoration: const InputDecoration(labelText: 'Description'),
                maxLines: 3,
              ),
              const SizedBox(height: 24),
              ElevatedButton(onPressed: _submit, child: const Text('Save Activity')),
            ],
          ),
        ),
      ),
    );
  }
}
KEY POINTS
  • This screen integrates Unit 18 (Forms), Unit 20 (context.read for a one-off write), and the native showDatePicker dialog in one cohesive form.
  • context.read<T>(), not watch<T>(), is correct inside _submit() — it's a one-off action, not something that should rebuild the screen on every model change.

Step 7The Details Screen & Deletion

class DetailsScreen extends StatelessWidget {
  final Activity activity;
  const DetailsScreen({super.key, required this.activity});

  @override
  Widget build(BuildContext context) {
    return Scaffold(
      appBar: AppBar(
        title: Text(activity.title),
        actions: [
          IconButton(
            icon: const Icon(Icons.delete),
            onPressed: () async {
              final confirmed = await showDialog<bool>(
                context: context,
                builder: (context) => AlertDialog(
                  title: const Text('Delete this activity?'),
                  actions: [
                    TextButton(
                      onPressed: () => Navigator.pop(context, false),
                      child: const Text('Cancel'),
                    ),
                    TextButton(
                      onPressed: () => Navigator.pop(context, true),
                      child: const Text('Delete'),
                    ),
                  ],
                ),
              );
              if (confirmed == true && context.mounted) {
                context.read<ActivityListModel>().deleteActivity(activity.id!);
                Navigator.pop(context); // back to the home list
              }
            },
          ),
        ],
      ),
      body: Padding(
        padding: const EdgeInsets.all(16),
        child: Column(
          crossAxisAlignment: CrossAxisAlignment.start,
          children: [
            Text(activity.category, style: const TextStyle(color: Colors.grey)),
            const SizedBox(height: 8),
            Text('${activity.date.toLocal()}'.split(' ').first),
            const SizedBox(height: 16),
            Text(activity.description),
          ],
        ),
      ),
    );
  }
}
CAUTIONThe context.mounted check after the await showDialog(...) call guards against calling Navigator/context.read on a BuildContext that may have been removed from the tree while the dialog was open — exactly the caution flagged in Unit 15 about using BuildContext after an await.
KEY POINTS
  • showDialog<bool> returning a typed result is the same await-and-pop pattern from Unit 19, applied to a confirmation dialog instead of a full screen.
  • Checking context.mounted after an await is a small habit that prevents an entire class of "used after dispose" runtime errors.

Step 8Testing the Capstone App

Following Unit 26, the model's logic is verified with a fast unit test using a fake repository — no real database needed — and the add-activity flow is verified with a widget test.

class FakeActivityRepository implements ActivityRepository {
  final List<Activity> _store = [];
  int _nextId = 1;

  @override
  Future<List<Activity>> getAll() async => List.from(_store);

  @override
  Future<Activity> insert(Activity activity) async {
    final saved = activity.copyWith(id: _nextId++);
    _store.insert(0, saved);
    return saved;
  }

  @override
  Future<void> delete(int id) async {
    _store.removeWhere((a) => a.id == id);
  }
}

void main() {
  test('adding an activity increases totalCount', () async {
    final model = ActivityListModel(FakeActivityRepository());
    await model.load();
    expect(model.totalCount, 0);

    await model.addActivity(Activity(
      title: 'FDP', category: 'Workshop',
      date: DateTime.now(), description: 'Test',
    ));

    expect(model.totalCount, 1);
  });
}
KEY POINTS
  • Because ActivityListModel depends on the abstract shape of a repository rather than sqflite directly, testing it needs no real database at all — exactly the payoff promised back in Unit 24.
  • This fake-repository technique scales to every layer of the app: the same idea would let you test AddActivityScreen's widget behaviour without touching real persistence either.

Step 9Capstone Summary & Next Steps

This nine-step build touched the model layer, a sqflite repository, Provider-based state management, forms with validation, navigation with results, dialogs, lifecycle-safe async handling, and testing with fakes — the same set of concerns present in almost any production Flutter app, just at a larger scale.

Extending This Project

  • Add search/filter by category using the where techniques from Unit 5 and the Iterables deep dive.
  • Add a REST backend and swap ActivityRepository's implementation for one using http (Unit 21) — the UI and state layers would not need to change at all, demonstrating the real value of the repository pattern.
  • Add a chart summarizing activities by category using a charting package, reusing the same ActivityListModel data.
  • Package it with a proper app icon, splash screen, and release build following Unit 27 and the App Icons/Splash unit before sharing it with others.
TIPThe fact that swapping local storage for a network API would only require changing one class — the repository — is not an accident. It is the entire point of the layered architecture introduced in Unit 24 and used consistently throughout this capstone.
KEY POINTS
  • A small, complete app touches nearly every concept in this manual — the scale changes in real production apps, but the underlying patterns do not.
  • Good layering (model / data / state / UI) is what makes an app's future changes — a new backend, a new feature, a new UI — cheap rather than a rewrite.
